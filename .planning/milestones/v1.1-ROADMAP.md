# Milestone v1.1: Colleague Lookup

**Status:** SHIPPED 2026-03-25
**Phases:** 10-12
**Total Plans:** 9

## Overview

Enable colleague search and profile display via Microsoft Graph API, rendered as inline profile cards in the chat UI.

## Phases

### Phase 10: Graph Client Foundation

**Goal**: A verified, isolated Graph API client exists with confirmed admin consent, correct token acquisition, and all three core operations tested in isolation
**Depends on**: Phase 9 (v1.0 shipped — additive work)
**Requirements**: GRAPH-01, GRAPH-02, GRAPH-03, SRCH-04
**Plans**: 4 plans

Plans:

- [x] 10-01: Azure AD app permissions + admin consent verification (GRAPH-02)
- [x] 10-02: graph_client.py — MSAL singleton, token acquisition, caching (GRAPH-01, GRAPH-03)
- [x] 10-03: User search and profile methods with ConsistencyLevel header (SRCH-04)
- [x] 10-04: Unit tests for graph_client.py core operations (gap closure)

**Details:**
- MSAL ConfidentialClientApplication singleton isolated from SSO auth
- Module-level token caching with automatic refresh
- search_users() with ConsistencyLevel: eventual header
- get_user_photo_bytes() returns None (not exception) on missing photo
- 10 pytest unit tests with fully mocked HTTP

**Goal**: Two new MCP tools are registered and callable, and authenticated users can retrieve colleague photos through a secure Flask proxy that absorbs 404s
**Depends on**: Phase 10 (graph_client.py verified in isolation)
**Requirements**: MCP-01, MCP-02, SRCH-01, SRCH-02, SRCH-03, PROF-01, PROF-03, PROF-05
**Plans**: 3 plans

Plans:

- [x] 11-01: MCP tool definitions in tools.py — search_colleagues and get_colleague_profile schemas (MCP-01, MCP-02)
- [x] 11-02: Tool dispatch + server.py graph client singleton initialization (SRCH-01, SRCH-02, SRCH-03, PROF-01, PROF-05)
- [x] 11-03: Photo proxy route in app.py with @login_required and 404 absorption (PROF-03)

**Details:**
- asyncio.to_thread wrapping sync graph_client calls in async MCP handlers
- Lazy import of graph_client inside handler bodies
- photo_url proxy pattern: /api/photo/{user_id} — binary data stays out of tool results
- SVG placeholder with initials for users without photos
- TOOL_DISPATCH extended from 15 to 17 entries

## Milestone Summary

**Key Decisions:**

- Use msal + requests directly — msgraph-sdk rejected (7 new transitive packages for two REST endpoints)
- photo_url proxy indirection — binary photo data never enters LLM context
- asyncio.to_thread for sync graph_client calls in async MCP handlers
- Lazy graph_client imports in handler bodies — avoids Config evaluation at module import time
- System prompt rules 7-10 numbered as continuation of 1-6
- Rule 10 explicitly names UI behavior to justify model brevity

**Issues Resolved:**

- Admin consent for User.Read.All and ProfilePhoto.Read.All granted
- Token layer implemented with automatic refresh
- Photo 404 absorption with SVG placeholder fallback

**Issues Deferred:**

- search_colleagues description phrasing ("Use for" vs "Use when") — test regression
- get_colleague_profile missing example query in description — test regression
- test_list_tools_returns_all_15 asserts 15 but should be 17 — stale assertion
- get_colleague_profile user_id schema says "user ID" but search returns email — misleading
- get_user_photo_bytes() dead code (get_user_photo_96 used instead)

**Technical Debt Incurred:**

- 3 test regressions from new tool descriptions not matching existing test conventions
- 1 orphaned function (get_user_photo_bytes) — may be useful for future non-96px sizes
