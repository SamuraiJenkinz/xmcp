Create `frontend/src/hooks/useStreamingMessage.ts`. This is a direct port of the SSE streaming logic from `chat_app/static/app.js` (the `readSSEStream` function and related state) into a React hook.

**Import types from `../types`:** SSEEvent, ToolPanelData. The hook will accept callback functions rather than directly importing contexts (keeps it testable).

**Hook signature:**
```typescript
interface UseStreamingMessageOptions {
  onText: (delta: string) => void;          // Called with batched text (rAF)
  onTool: (event: ToolPanelData) => void;   // Called per tool event
  onThreadNamed: (threadId: number, name: string) => void;
  onDone: () => void;
  onError: (message: string) => void;
  onCancel: () => void;
}

export function useStreamingMessage(options: UseStreamingMessageOptions): {
  startStream: (message: string, threadId: number) => void;
  cancelStream: () => void;
  isStreaming: boolean;
}
```

**AbortController:** Store in `useRef<AbortController | null>(null)`. NEVER in useState. This is a locked decision from research.

**rAF text batching (port from research Pattern 5):**
- `pendingTextRef = useRef('')` accumulates text deltas
- `rafRef = useRef<number | null>(null)` tracks the rAF ID
- On each `text` SSE event: append delta to pendingTextRef. If no rAF scheduled, schedule one that flushes pendingTextRef via `options.onText(pendingTextRef.current)` then clears it.
- On finalize/cancel: cancel any pending rAF with `cancelAnimationFrame`, flush remaining text.

**SSE buffer parsing (port from app.js readSSEStream verbatim):**
```typescript
const decoder = new TextDecoder('utf-8');
let buffer = '';

// In the read loop:
buffer += decoder.decode(value, { stream: true });  // { stream: true } is CRITICAL
const parts = buffer.split('\n\n');
buffer = parts.pop() ?? '';
for (const block of parts) {
  for (const line of block.split('\n')) {
    if (line.startsWith('data: ')) {
      const json = line.slice(6);
      const event: SSEEvent = JSON.parse(json);
      handleEvent(event);
    }
  }
}
```

**Event handling:**
- `text` → append delta to pendingTextRef, schedule rAF if needed
- `tool` → flush any pending text first, then call `options.onTool({ name, params, result, status })`
- `thread_named` → call `options.onThreadNamed(event.thread_id, event.name)`
- `done` → flush pending text, call `options.onDone()`, set isStreaming=false
- `error` → flush pending text, call `options.onError(event.message)`, set isStreaming=false

**startStream function:**
1. Create new AbortController, store in ref
2. Set isStreaming = true (use useState for this one — it controls UI)
3. Call `fetch('/chat/stream', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ message, thread_id: threadId }), signal })`
4. On response: check `res.ok`, then enter the ReadableStream read loop
5. On AbortError in catch: call `options.onCancel()`
6. On other errors in catch: call `options.onError(err.message)`

**cancelStream function:**
1. `abortControllerRef.current?.abort()`
2. `abortControllerRef.current = null`
3. Cancel any pending rAF
4. Flush remaining text

**Cleanup:** Use `useEffect` cleanup to abort any in-flight stream on unmount. Cancel any pending rAF.

**Use `useCallback` for startStream and cancelStream** to ensure stable references. Include `options` in a ref to avoid stale closures (the options callbacks will change as contexts update).

Create `frontend/src/utils/parseHistoricalMessages.ts`. This is the historical message parsing algorithm from the research (14-RESEARCH.md, "DEBT-01 Deep Dive" section).

**Key insight from research:** DEBT-01 is a frontend-only fix. Tool messages (role: "assistant" with tool_calls + role: "tool") are ALREADY persisted in `messages_json`. The backend `_message_to_dict()` in openai_client.py preserves `tool_calls`. The fix is parsing them correctly when loading history.

**Function signature:**
```typescript
import { RawMessage, DisplayMessage, ToolPanelData } from '../types';

export function parseHistoricalMessages(raw: RawMessage[]): DisplayMessage[]
```

**Algorithm (direct from research):**
```
For each message in raw (iterate with index):
  if role === 'system' → skip
  if role === 'user' → push { type: 'user', content: msg.content ?? '' }
  if role === 'assistant':
    - Collect toolPanels: if msg.tool_calls exists, for each tool_call:
      - Parse params from tool_call.function.arguments (try/catch JSON.parse, default to {})
      - Look ahead for matching role:'tool' message where tool_call_id === tc.id
      - Extract result from the tool message's content
      - Push { name: tc.function.name, params, result, status: 'success' } to toolPanels
    - If msg.content is non-null and non-empty:
      - Push { type: 'assistant', content: msg.content, toolPanels }
    - If msg.content is null/empty but toolPanels exist:
      - Do NOT render this as a standalone message — the tool panels will attach
        to the NEXT assistant message that has content (this handles the multi-turn
        tool loop where assistant messages alternate between tool_calls-only and
        final-text-only)
  if role === 'tool' → skip (consumed by assistant tool_calls pairing above)
```

**Important refinement for multi-turn tool loops:** When multiple tool loop iterations occur, there can be multiple assistant(tool_calls) + tool(result) pairs before the final assistant(content). Accumulate ALL tool panels across the loop and attach them to the final content-bearing assistant message.

**Updated algorithm that handles accumulation:**
```typescript
const result: DisplayMessage[] = [];
let pendingToolPanels: ToolPanelData[] = [];
let i = 0;

while (i < raw.length) {
  const msg = raw[i];

  if (msg.role === 'system') { i++; continue; }

  if (msg.role === 'user') {
    result.push({ type: 'user', content: msg.content ?? '' });
    i++;
    continue;
  }

  if (msg.role === 'assistant') {
    if (msg.tool_calls) {
      // Collect tool panels from this assistant+tool pair
      for (const tc of msg.tool_calls) {
        let params: Record<string, unknown> = {};
        try { params = JSON.parse(tc.function.arguments); } catch {}

        let toolResult: string | null = null;
        // Look ahead for matching tool result
        for (let j = i + 1; j < raw.length; j++) {
          if (raw[j].role === 'tool' && raw[j].tool_call_id === tc.id) {
            toolResult = raw[j].content ?? '';
            break;
          }
          if (raw[j].role === 'assistant') break;
        }

        pendingToolPanels.push({
          name: tc.function.name,
          params,
          result: toolResult,
          status: 'success',
        });
      }
    }

    if (msg.content) {
      // This is a content-bearing assistant message — attach all accumulated tool panels
      result.push({
        type: 'assistant',
        content: msg.content,
        toolPanels: pendingToolPanels.length > 0 ? [...pendingToolPanels] : undefined,
      });
      pendingToolPanels = [];
    }
    i++;
    continue;
  }

  if (msg.role === 'tool') { i++; continue; }

  i++;  // Skip unknown roles
}

return result;
```

This correctly handles:
- Simple messages (user + assistant with content only)
- Single tool call (assistant with tool_calls → tool → assistant with content)
- Multi-turn tool loops (multiple assistant(tool_calls) + tool pairs → final assistant with content)
- Edge cases: empty content, missing tool results
