<objective>
Refine and validate all 15 Exchange tool descriptions for LLM tool-selection accuracy. Each description must be unambiguous enough that an LLM consistently picks the correct tool for a given natural language query.

Purpose: Tool descriptions are the primary signal an LLM uses to select which tool to call. Vague or overlapping descriptions cause wrong tool selection, wasted API calls, and incorrect answers. This plan treats descriptions as a first-class concern separate from the registration infrastructure.

Output: Refined tool descriptions in tools.py and a dedicated test file that validates description quality properties (length, uniqueness, plain language, example queries).
</objective>

<task type="auto">
  <name>Task 1: Refine tool descriptions for disambiguation and LLM accuracy</name>
  <files>exchange_mcp/tools.py</files>
  <action>
Review and refine each of the 15 Exchange tool descriptions in TOOL_DEFINITIONS. Apply these rules systematically:

**Structure for each description** (aim for this pattern):
1. First sentence: What the tool returns (be specific about the data fields)
2. Second sentence: "Use when asked about [specific triggers]"
3. Third sentence: 2-3 example natural language queries in quotes
4. Optional: One sentence clarifying what this tool does NOT do (for disambiguation from similar tools)

**Critical disambiguation pairs to address:**

A. **get_mailbox_stats vs search_mailboxes:**
   - get_mailbox_stats: ONE specific user's mailbox details (size, quota, last login, database)
   - search_mailboxes: FIND/LIST multiple mailboxes matching a filter (by database, type, or name pattern)
   - The description for get_mailbox_stats should say "for a single user" and "Use when asked about ONE person's mailbox"
   - The description for search_mailboxes should say "find or list mailboxes" and "Use when asked to find, list, or search for mailboxes"

B. **get_shared_mailbox_owners vs search_mailboxes:**
   - get_shared_mailbox_owners: WHO has access to a shared mailbox (delegates/permissions)
   - search_mailboxes: FIND shared mailboxes (by type filter)
   - The description for get_shared_mailbox_owners should mention "permissions", "access", "delegates", "full access", "send-as", "send-on-behalf"

C. **get_dmarc_status vs get_dkim_config:**
   - get_dmarc_status: DMARC policy AND SPF record for a domain (DNS-based, no PowerShell)
   - get_dkim_config: DKIM signing configuration and CNAME records (Exchange PowerShell-based)
   - get_dmarc_status description should mention "DMARC policy, SPF record, email authentication overview"
   - get_dkim_config description should mention "DKIM signing, CNAME records, signing domains"

D. **list_dag_members vs get_dag_health:**
   - list_dag_members: Server inventory (which servers are in the DAG, operational status)
   - get_dag_health: Replication health (queue lengths, content index state, copy status)

E. **get_hybrid_config vs get_connector_status:**
   - get_hybrid_config: Full hybrid topology overview (org relationships, federation)
   - get_connector_status: Health check of hybrid connectors (live test against Exchange Online)

**Plain language rules:**
- "email address" not "UPN" or "identity" or "mailbox identity"
- "user" or "person" not "recipient" or "mailbox-enabled user"
- "database availability group" or "DAG" (first use expanded, subsequent uses can say DAG)
- "email server" not "transport server" or "hub transport"
- "send connectors and receive connectors" not "SMTP connectors" alone
- "email routing" not "message routing" or "transport routing"

**Length constraint:** Each description MUST be under 800 characters. Aim for 300-600 characters (the sweet spot for LLM tool selection).

After refining descriptions, verify: read through all 15 descriptions in sequence and confirm no two descriptions could plausibly match the same natural language query.

<task type="auto">
  <name>Task 2: Write description quality tests</name>
  <files>tests/test_tool_descriptions.py</files>
  <action>
Create tests/test_tool_descriptions.py with tests that validate tool description quality properties. These tests act as a regression guard -- if someone changes a description in a way that degrades quality, the test fails.

Tests to write:

1. **test_all_descriptions_under_800_chars** -- Iterate TOOL_DEFINITIONS (excluding ping), assert len(tool.description) <= 800 for each. Fail message should include the tool name and actual length.

2. **test_all_descriptions_contain_use_when** -- Every Exchange tool description (excluding ping) contains "Use when" (case-insensitive). This ensures each description has a trigger phrase for the LLM.

3. **test_all_descriptions_contain_example_query** -- Every Exchange tool description (excluding ping) contains at least one quoted example query (look for single or double quotes containing a question-like phrase, or phrases starting with "how", "what", "who", "when", "list", "find", "check", "show"). A simple heuristic: description contains at least one of the example trigger words.

4. **test_no_exchange_jargon** -- Check that no description contains forbidden jargon terms: "UPN", "identity" (as a standalone word, not inside "identify"), "recipient object", "mailbox-enabled", "cmdlet", "PowerShell", "ConvertTo-Json". Use regex word-boundary matching for "identity" to avoid false positives.

5. **test_tool_names_are_snake_case** -- All tool names match the pattern `^[a-z][a-z0-9_]*$`.

6. **test_no_duplicate_tool_names** -- All tool names are unique.

7. **test_input_schemas_are_valid_json_schema** -- Every tool's inputSchema has "type": "object" and "properties" as a dict. Every property within has "type" and "description".

8. **test_required_params_exist_in_properties** -- For every tool, each item in inputSchema["required"] must be a key in inputSchema["properties"].

9. **test_mailbox_stats_vs_search_disambiguation** -- get_mailbox_stats description contains "single" or "one" (singular user), search_mailboxes description contains "find" or "list" or "search" (multiple results). They should NOT both contain the same trigger phrases.

10. **test_dmarc_vs_dkim_disambiguation** -- get_dmarc_status description contains "DMARC" and "SPF", get_dkim_config description contains "DKIM" and "signing" or "CNAME". They should have distinct trigger phrases.
